
## Plan specs

For major changes, the **first step of planning** is to write a plan spec to [docs/](docs/) as `<slug>-spec.md`. That file — not any ephemeral internal plan — is the source of truth for the work. Keep it in sync as the approach evolves, and reference it from commit messages / PR descriptions.

Spec contents:
- **Status** and date
- **Motivation** — why this change, what problem it solves
- **Non-goals** — what is explicitly out of scope
- **Approach** — the chosen design, with links to the existing code paths it touches
- **Changes** — concrete files/areas being modified
- **Verification** — how to confirm the change works end-to-end
- **Rollback** — how to undo if it doesn't pan out

## Order of operations
Explain all file edits thoroughly BEFORE asking for user approval
